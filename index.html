Scaling Strategy - Phase 1

GraalVM Native + Warm Pool Approach

Native Compilation Benefits

  • GraalVM Native Images: Compile Java applications to native executables
  • Reducing startup from 2-3 minutes to 15-30 seconds

Warm Pool Configuration

  • Baseline: 3 pre-warmed pods during normal traffic
  • Memory: 128Mi request, 256Mi limit (50% reduction vs JVM)
  • CPU: 100m request, 500m limit
  • Cost: ~$75-150/month for 24/7 warm pool

Scale-Down Strategy & Graceful Termination

Phase 1: Preparation (0-10 seconds)

  • Pod receives SIGTERM signal
  • Update readiness probe to fail
  • Run preStop hooks and cleanup

Phase 2: Graceful Completion (10-60 seconds)

  • Complete in-flight requests
  • Close connections and flush caches
  • Ensure data consistency

Phase 3: Force Termination

  • SIGKILL if grace period exceeded
  • Monitor and alert on forced terminations

Cool Down Configuration

  • HPA Stabilization: 300s window to prevent thrashing
  • KEDA Cool Down: 300s for event-driven scalers
  • Custom Metrics: Business-logic-based cool-down

Architecture Patterns - Resilience

Circuit Breaker Pattern

Implementation: Use Spring Cloud Circuit Breaker with fallback methods to prevent cascade failures and provide graceful degradation during database overload

Configuration

  • Failure Threshold: 50% failures in 10 requests
  • Wait Duration: 30 seconds before retry
  • Fallback: Cached data or graceful degradation

CQRS (Command Query Responsibility Segregation)

Strategy: Split read and write operations into separate services allowing independent scaling

Scaling Strategy

  • Command Services: 2-5 replicas (writes are lower volume)
  • Query Services: 5-125 replicas (25x scaling for reads)
  • Event Streaming: Kafka handles async communication

Data Layer Scaling & Caching Strategy

Database Connection Pooling

HikariCP Configuration

  • Dynamic Scaling: 10 (normal) to 50 (spike) connections per service
  • Pool Settings: maximumPoolSize=50, minimumIdle=10
  • Multi-tier Pooling: Separate pools for critical vs non-critical services

Read Replica Architecture

  • Multi-AZ Replicas: Deploy 3-5 read replicas across availability zones
  • Read Scaling: Handle 80% of read traffic through replicas during normal operations, 90% during spikes
  • Connection String Management: Environment variables for primary and replica connections
